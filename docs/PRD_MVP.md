# 電子寵物任務系統 PRD（當前實作對照版）

## 1. 文檔目的

這份 PRD 不是重新發想版本，而是用來對照目前專案已完成內容、已知設計方向、以及尚未定義清楚的地方。  
如果你想檢查「是不是原始 PRD 有問題」，最重要的是看：

- 哪些需求已經能穩定落地
- 哪些需求只有概念，還沒有精確規格
- 哪些功能其實跨了兩個不同系統層次
  - Web 產品層
  - 本地素材流水線層

## 2. 產品目標

### 核心目標

建立一個家庭可實際使用的電子寵物任務系統，讓家長可以透過任務派發與審核機制，驅動孩子完成日常事項，並用電子寵物狀態作為回饋。

### 核心成功指標

- 家長能穩定派發任務並完成審核
- 孩子能理解今日要做什麼，並完成提交
- 寵物狀態能正確反映孩子當日進度
- 權限隔離清楚，不會看見其他家庭資料

### 次階段成功指標

- 寵物角色資產具備一致性
- 素材可穩定批量生成
- 後續可以擴展道具、動畫、換裝與更多角色

## 3. 目標使用者

### 家長

- 建立孩子帳號
- 派發任務
- 審核孩子提交
- 查看每日完成情況

### 孩子

- 看今天的任務與寵物狀態
- 提交已完成任務
- 從寵物回饋中感受成就

### 管理員

- 管理家長 / 孩子帳號
- 協助重設密碼
- 協助排查綁定與權限問題

## 4. 產品邊界

### 本 PRD 內

- Web 產品流程
- 權限與帳號模型
- 任務審核制
- 寵物狀態規則
- 商店與道具購買
- invite onboarding
- 家庭資料隔離
- 本地素材資產工程

### 本 PRD 暫不包含

- 真正上架的 App 版本
- 雲端多租戶運維
- 推播通知
- 複雜社交機制
- 排行榜
- 即時動畫引擎

## 5. 核心使用流程

### 流程 A：家長建立與綁定孩子

1. 家長註冊並設定密碼
2. 家長建立 invite link
3. 孩子透過 `/login?invite=TOKEN` 進入 onboarding
4. 孩子完成基本資料、角色設定、寵物命名與密碼設置
5. 系統完成 parent-child 關聯

### 流程 B：家長派發與審核

1. 家長到 `/parent`
2. 選擇孩子
3. 批量派發今日任務
4. 孩子完成後提交
5. 家長 approve / reject
6. 只有 approved 任務進入 food / coin 計分

### 流程 C：孩子端回饋

1. 孩子登入 `/child`
2. 查看寵物狀態與今日任務
3. 對 pending 任務點「我完成了」
4. 任務進入 submitted
5. 家長審核後，寵物狀態變化

## 6. 任務規則

### 任務類型

- 主線：貓糧
- 支線：貓星幣

### 任務狀態

- `pending`
- `submitted`
- `approved`
- `rejected`

### 計分規則

- 主線 approved 數 × 10 = food
- 支線 approved 數 × 10 = coin
- submitted 不計分
- rejected 不計分

### 每日規則

- 每日最多 10 項任務
- threshold 來自 `app_config`
- food 低於 threshold 時寵物為 `hungry`
- 達 threshold 後依完成率決定 mood

## 7. 權限模型

### role

- `admin`
- `parent`
- `child`

### 關聯規則

- `child.parent_user_id` 指向其所屬家長
- 家長只能查詢自己名下孩子
- 管理員可跨家庭查看資料

### 安全原則

- child 不可 approve / reject
- parent 不可越權查看他人孩子
- 被停用帳號不可登入

## 8. 寵物系統需求

### 產品層需求

- 孩子帳號綁定一個寵物角色
- 寵物具備：
  - `petStatus`
  - `mood`
  - `todayFood`
  - `todayCoin`
- 頁面上顯示角色圖、狀態文案、道具結果

### 資產層需求

- 寵物不是一張臨時圖，而是素材庫角色
- 多個孩子可以共用同一角色
- 狀態圖需能對應：
  - `calm`
  - `happy`
  - `hungry`
  - `angry`
  - `sleepy`
  - `excited`

## 9. 商店與道具

### 當前需求

- 用 coin 購買內建道具
- 查看已購買清單

### 尚未完全定義的部分

- 道具是否只是收藏，還是可以裝備
- 若可裝備，slot 如何定義
- 道具是否會影響角色顯示
- 動態角色與道具掛點如何處理

這一塊目前是 PRD 的典型模糊區。

## 10. 素材與角色一致性

### 目前正確方向

- 角色屬於素材庫，不屬於單一孩子帳號
- 孩子只是綁定某個素材角色
- 後續應優先保證：
  - 同一角色在不同狀態下仍像同一隻貓
  - 狀態差異不只靠表情，而要靠輪廓與姿態

### 目前已建立的工程基礎

- `pet_pipeline/`
- ComfyUI production / inpaint
- Ollama 本地視覺 QC
- OpenClaw skill 調度

## 11. 非功能需求

### 穩定性

- child / parent 端防重入
- 非法狀態流轉回 `409`
- child hidden 狀態暫停輪詢

### 可維護性

- 角色 / 狀態 / profile 配置檔化
- 流程可重複跑
- 素材流水線與 Web 主專案解耦

### 可擴展性

- 可新增更多角色
- 可新增更多狀態
- 可新增 LoRA 訓練與量產流程

## 12. 目前 PRD 最可能有問題的地方

這一段是目前最值得你回頭檢查的。

### 問題 1：把「產品需求」和「素材生產需求」混在一起

目前有兩種需求常被說成一件事：

- Web 產品要做什麼
- 角色素材要怎麼做

這兩層如果不拆開，後面會一直出現：

- 產品已經能跑，但素材還不夠
- 素材工作流已建，但產品還沒定好展示方式

建議 PRD 之後拆成：

- Product PRD
- Asset / Character Pipeline Spec

### 問題 2：狀態需求沒有明確區分「表情型」和「姿態型」

例如：

- `calm` 可以主要靠表情
- `happy / angry / hungry / excited` 其實是姿態型狀態

如果 PRD 沒寫清楚，技術就很容易先用「同一坐姿 + 改臉」硬做，結果看起來不像完整狀態。

### 問題 3：道具需求不夠精確

目前只定義到：

- 可買道具
- 可查看已購

但還沒清楚定義：

- 是否能裝備
- 是否影響 UI
- 是否是 overlay
- 是否跟動畫掛點綁定

如果這個不早點定義，後續很容易返工。

### 問題 4：角色資產一致性的驗收標準沒有明文化

你現在最在意的是：

- 玩家能不能一眼認出是同一隻貓

但若 PRD 沒把這件事寫成驗收標準，產圖會很容易回到「單張好看但不像同一角色」。

建議明確補上：

- identity consistency 為資產第一優先
- 表情 / 輪廓差異為第二優先
- 美術發散為第三優先

### 問題 5：部署策略還沒在 PRD 裡正式落位

現在你已經明確方向是：

- MVP 先 Web / H5
- 家長 / 小孩手機使用
- 管理員用 Web

這應該被正式寫進 PRD，而不是只停留在對話決策裡。

## 13. 建議下一版 PRD 補充的決策欄位

建議你下一版 PRD 至少補這幾個欄位：

- 寵物狀態表：哪些是表情型、哪些是姿態型
- 道具系統：收藏 / 裝備 / slot / 視覺作用
- 角色資產驗收標準
- 動畫是否列入 MVP
- Web / H5 是否為正式第一階段交付
- 本地素材流水線是否列為內部生產系統

## 14. 當前建議的產品優先順序

1. 穩定 Web MVP 主流程
2. 完成角色資產第一輪一致性驗證
3. 補孩子選寵物
4. 補道具裝備展示
5. 再評估動畫與 App 化

## 15. 結論

目前這個項目最大的風險，不是程式架構，而是 PRD 裡有幾個跨層級需求還沒被切清楚：

- 產品流程
- 角色素材規格
- 道具視覺系統
- 動畫與後續 App 方向

如果你想快速判斷 PRD 哪裡有問題，優先看：

1. 狀態需求是否區分表情型 / 姿態型
2. 道具是否定義成收藏還是裝備
3. 角色一致性是否成為正式驗收標準
4. Web/H5 是否被正式列為第一階段方案
